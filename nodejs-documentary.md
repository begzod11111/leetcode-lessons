# Документальный гид: Node.js, V8, libuv и event loop

> В запросе встречаются названия "LiveWiWi", "E8" и "NJS". Ниже я использую
> общепринятые термины, которые подходят к этим темам: **libuv**, **V8** и
> **Node.js**.

## 1. О чем этот фильм

Представьте, что JavaScript-программа - это маленькая съемочная группа. Главный
актер говорит только на JavaScript. Оператор снимает каждую сцену, режиссер
решает, что делать дальше, а большая техническая команда за кадром работает с
файлами, сетью, таймерами и потоками.

В Node.js роли распределены так:

- **V8** исполняет JavaScript: парсит код, оптимизирует горячие функции,
  управляет памятью и запускает garbage collector.
- **Node.js** дает JavaScript доступ к системным возможностям: файлам, сети,
  процессам, потокам, переменным окружения и модулям.
- **libuv** связывает Node.js с операционной системой: event loop, таймеры,
  неблокирующий ввод-вывод, thread pool.
- **Операционная система** реально умеет ждать события от сокетов, файлов,
  терминала и таймеров через механизмы вроде `epoll`, `kqueue`, `IOCP`.

Главная идея: JavaScript в Node.js обычно выполняется в одном основном потоке,
но ожидание медленных операций выносится за пределы этого потока. Поэтому
программа может обслуживать много запросов, не создавая отдельный поток на
каждый запрос.

## 2. С чего начать Node.js

Минимальная программа:

```js
console.log("Hello, Node.js");
```

Запуск:

```bash
node app.js
```

Базовые понятия:

1. **Файл `.js`** - обычный JavaScript-код.
2. **Модуль** - отдельный файл или пакет, который можно импортировать.
3. **npm** - менеджер пакетов JavaScript.
4. **`package.json`** - паспорт проекта: имя, команды, зависимости.
5. **Асинхронность** - стиль, при котором программа начинает операцию и
   продолжает жить, пока результат готовится.

Пример асинхронного чтения файла:

```js
const fs = require("node:fs/promises");

async function main() {
  const text = await fs.readFile("README.md", "utf8");
  console.log(text);
}

main().catch(console.error);
```

Для старта полезно уверенно понимать:

- переменные, функции, объекты, массивы;
- callback, Promise, `async`/`await`;
- отличие синхронных и асинхронных функций;
- импорт модулей;
- обработку ошибок через `try/catch` и `.catch()`.

## 3. V8: двигатель JavaScript

**V8** - движок JavaScript, созданный Google. Он используется в Chrome и
Node.js. Его задача - взять текст JavaScript и сделать так, чтобы процессор
выполнил нужные инструкции.

Упрощенный путь кода:

1. **Parsing** - V8 читает текст и строит внутреннее представление программы.
2. **Bytecode** - интерпретатор Ignition превращает код в байткод.
3. **Execution** - байткод выполняется.
4. **Optimization** - если функция часто вызывается, TurboFan может
   оптимизировать ее в более быстрый машинный код.
5. **Deoptimization** - если предположения оптимизатора стали неверными,
   функция может вернуться к менее оптимизированному режиму.

Важно: V8 не является всем Node.js. Он не занимается сетевыми сокетами,
файловой системой и event loop целиком. Он отвечает за JavaScript и память
JavaScript. Остальное добавляет Node.js и libuv.

### Call stack

Call stack - это стек вызовов функций:

```js
function c() {
  console.log("inside c");
}

function b() {
  c();
}

function a() {
  b();
}

a();
```

Когда вызывается `a`, она попадает в стек. Потом `b`, потом `c`. Когда `c`
заканчивается, она выходит из стека, затем `b`, затем `a`.

Если стек занят долгой синхронной работой, event loop не может продолжить
обработку других событий:

```js
while (true) {
  // Бесконечный цикл блокирует основной поток.
}
```

## 4. Event loop: сердце асинхронности

Event loop - это цикл, который постоянно спрашивает:

1. Есть ли готовые таймеры?
2. Есть ли завершенные операции ввода-вывода?
3. Есть ли callbacks, которые нужно вызвать?
4. Есть ли микрозадачи Promise?
5. Можно ли снова перейти в ожидание событий?

Пример:

```js
console.log("1");

setTimeout(() => {
  console.log("2: timer");
}, 0);

Promise.resolve().then(() => {
  console.log("3: promise");
});

console.log("4");
```

Обычно результат будет:

```text
1
4
3: promise
2: timer
```

Почему:

- обычный синхронный код идет первым;
- Promise callback попадает в очередь микрозадач;
- `setTimeout` попадает в очередь таймеров;
- микрозадачи выполняются раньше перехода к следующей фазе event loop.

### Фазы event loop в Node.js

Упрощенно в Node.js есть такие фазы:

1. **timers** - callbacks от `setTimeout` и `setInterval`;
2. **pending callbacks** - некоторые системные callbacks;
3. **idle, prepare** - внутренние фазы libuv;
4. **poll** - ожидание и обработка I/O-событий;
5. **check** - callbacks от `setImmediate`;
6. **close callbacks** - закрытие ресурсов, например сокетов.

Отдельно Node.js обрабатывает:

- **microtask queue** для Promise;
- **`process.nextTick` queue**, которая имеет очень высокий приоритет и может
  "задушить" event loop, если использовать ее бесконечно.

## 5. libuv: команда за кадром

**libuv** - C-библиотека, на которой Node.js строит event loop и асинхронные
операции. Она дает единый интерфейс для разных операционных систем.

Node.js должен работать на Linux, macOS и Windows. Но у каждой системы свои
механизмы ожидания событий:

- Linux: `epoll`;
- macOS/BSD: `kqueue`;
- Windows: IOCP;
- также могут использоваться `poll` или `select` в некоторых случаях.

libuv прячет эти отличия. JavaScript-код вызывает `fs.readFile` или создает
сервер, а libuv выбирает, как договориться с ОС.

## 6. Event multiplexer и demultiplexer

Event multiplexer - это механизм, который позволяет одному потоку ждать много
источников событий сразу.

Без мультиплексора можно представить плохую модель:

```text
1 поток ждет 1 сокет
10000 клиентов = 10000 потоков
```

С мультиплексором:

```text
1 поток регистрирует много сокетов
ОС сообщает: "вот эти сокеты готовы"
event loop вызывает нужные callbacks
```

Термин **demultiplexer** часто означает часть, которая возвращает из общего
набора событий конкретные готовые события. Например:

1. приложение регистрирует интерес к сокету;
2. ОС ждет, пока сокет станет готов к чтению или записи;
3. demultiplexer возвращает список готовых событий;
4. event loop вызывает JavaScript callbacks.

Ключевой момент: readiness-событие не всегда означает "данные уже полностью
прочитаны". Оно означает "ресурс готов, можно попробовать читать или писать".

## 7. Как работают блокирующие операции и потоки

Не все операции ОС одинаково хорошо поддерживают неблокирующий режим. Например,
многие файловые операции удобнее выполнять в отдельном worker-потоке. Поэтому
libuv содержит **thread pool**.

Типичные операции, которые могут использовать thread pool:

- файловая система (`fs`);
- DNS-запросы некоторых типов;
- криптография (`crypto`);
- сжатие (`zlib`).

Схема:

```text
JavaScript вызывает async API
Node.js передает задачу libuv
libuv кладет задачу в thread pool
worker-поток выполняет блокирующую работу
готовый результат возвращается в event loop
callback/Promise продолжается в основном JavaScript-потоке
```

Важно: результат callback все равно выполняется в основном JS-потоке, если вы
не используете `worker_threads`.

### Синхронные API блокируют event loop

```js
const fs = require("node:fs");

const text = fs.readFileSync("README.md", "utf8");
console.log(text);
```

Пока `readFileSync` работает, основной поток занят. Для маленького скрипта это
может быть нормально. Для сервера это опасно: один медленный диск или большой
файл задержит всех клиентов.

### Worker Threads для CPU-bound задач

Thread pool libuv помогает с некоторыми системными операциями, но тяжелые
вычисления в JavaScript лучше выносить в `worker_threads`.

```js
const { Worker, isMainThread, parentPort } = require("node:worker_threads");

if (isMainThread) {
  const worker = new Worker(__filename);
  worker.on("message", console.log);
} else {
  let sum = 0;
  for (let i = 0; i < 1_000_000; i++) {
    sum += i;
  }
  parentPort.postMessage(sum);
}
```

Главное правило:

- I/O - чаще через async API Node.js;
- CPU-heavy задачи - через worker threads, отдельный процесс или внешнюю
  очередь задач.

## 8. Память в Node.js и V8

В JavaScript память в основном делится на:

- **stack** - стек вызовов и локальная служебная информация;
- **heap** - объекты, массивы, функции, замыкания;
- **external memory** - память за пределами обычного JS heap, например
  `Buffer` и некоторые native-ресурсы.

Пример объекта в heap:

```js
const user = {
  name: "Ada",
  roles: ["admin", "developer"],
};
```

Если на объект больше нет достижимых ссылок, garbage collector может освободить
его память.

### Garbage collector в JavaScript

Да, в JavaScript есть garbage collector. В Node.js им управляет V8.

Идея tracing GC:

1. GC начинает с корневых ссылок: глобальные переменные, стек, активные
   замыкания, внутренние ссылки V8.
2. Он помечает все достижимые объекты.
3. Все недостижимые объекты считаются мусором.
4. Память освобождается или уплотняется.

V8 использует поколенческий подход:

- **young generation** - новые объекты, которые часто быстро умирают;
- **old generation** - объекты, которые пережили несколько сборок мусора.

Это эффективно, потому что большинство временных объектов живет недолго.

### Почему память может утекать

Утечка в GC-языке обычно означает: объект уже не нужен человеку, но все еще
достижим для программы.

Частые причины:

- глобальные массивы и кэши без ограничения размера;
- listeners, которые не удаляются;
- таймеры, которые продолжают держать замыкания;
- Map, где ключи никогда не удаляются;
- большие Buffer-объекты;
- случайное хранение request/response объектов.

Пример риска:

```js
const cache = new Map();

function rememberUser(user) {
  cache.set(user.id, user);
}
```

Если `cache` растет бесконечно, GC не может удалить пользователей: на них все
еще есть ссылки.

## 9. Сравнение с Python

В Python тоже есть garbage collector, но детали отличаются.

В CPython основа управления памятью:

- **reference counting** - объект обычно освобождается, когда счетчик ссылок
  стал нулем;
- **cyclic garbage collector** - отдельный механизм для циклических ссылок;
- **GIL** - глобальная блокировка интерпретатора, влияющая на многопоточность
  Python-кода.

В V8:

- нет обычного deterministic reference counting как главного механизма;
- используется tracing/generational garbage collection;
- освобождение памяти не происходит строго в момент, когда вы "забыли" объект;
- GC может запускаться позже и иногда вызывать короткие паузы.

Общая мысль для Python и JavaScript одинаковая: если вы случайно сохраняете
ссылку на большой объект, сборщик мусора не удалит его.

## 10. Безопасность в Node.js

Node.js - мощная среда, потому что JavaScript получает доступ к ОС. Это удобно,
но означает, что безопасность зависит от кода, зависимостей и конфигурации.

### Что защищает, а что нет

- V8 изолирует выполнение JavaScript на уровне движка, но Node.js специально
  открывает доступ к файловой системе, сети и процессам.
- `vm` module полезен для некоторых задач, но не должен считаться надежной
  песочницей для враждебного кода.
- Зависимости из npm выполняются с правами вашего процесса.

### Основные риски

1. **Command injection**

   ```js
   const { exec } = require("node:child_process");

   exec(`convert ${userInput}`);
   ```

   Если `userInput` пришел от пользователя, он может содержать вредную команду.
   Безопаснее использовать `execFile`/`spawn` с массивом аргументов и строгой
   валидацией.

2. **Path traversal**

   ```js
   // Опасно, если name = "../../secret.txt"
   const path = `uploads/${name}`;
   ```

   Нужно нормализовать путь и проверять, что итоговый путь остается внутри
   разрешенной директории.

3. **Prototype pollution**

   Небезопасное глубокое объединение объектов может изменить `Object.prototype`
   и сломать логику приложения.

4. **Dependency risk**

   Любой пакет может иметь уязвимость или вредный install script.

5. **Denial of service**

   Бесконечные циклы, тяжелые regex, огромный JSON, слишком большие payloads и
   синхронные операции могут заблокировать event loop.

### Практический минимум безопасности

- валидировать все внешние данные;
- не доверять request body, headers, query params и env без проверки;
- ограничивать размер входящих данных;
- использовать async API вместо sync API на сервере;
- хранить секреты в переменных окружения, не коммитить их;
- обновлять зависимости;
- запускать процесс с минимальными правами;
- логировать ошибки без утечки секретов;
- использовать HTTPS и безопасные cookies для веб-приложений;
- держать boundary между доверенным и недоверенным кодом.

## 11. Мини-карта выполнения запроса в Node.js

Когда HTTP-запрос приходит в Node.js-сервер:

```text
клиент -> ОС -> event multiplexer -> libuv -> Node.js callback -> V8 выполняет JS
```

Если JavaScript запускает асинхронное чтение файла:

```text
JS вызывает fs.promises.readFile
Node.js/libuv передают работу в thread pool
основной JS-поток свободен
worker читает файл
результат возвращается в очередь event loop
Promise resolve
V8 продолжает async function
```

Если JavaScript делает тяжелый CPU-цикл:

```text
V8 занят
call stack не освобождается
event loop не может обрабатывать новые callbacks
сервер "зависает" для других клиентов
```

## 12. Хорошие привычки для фундамента

1. Понимать, когда код синхронный, а когда асинхронный.
2. Не блокировать event loop тяжелыми вычислениями.
3. Использовать Promise и `async`/`await` аккуратно, всегда обрабатывать ошибки.
4. Знать, что libuv не делает JavaScript многопоточным автоматически.
5. Отличать параллельность ожидания I/O от параллельного выполнения CPU-кода.
6. Следить за ростом памяти и кэшей.
7. Не запускать недоверенный код внутри своего Node.js-процесса.
8. Измерять, а не угадывать: `process.memoryUsage()`, profiler, heap snapshot,
   event loop delay.

## 13. Короткий глоссарий

- **V8** - JavaScript-движок, который исполняет код и управляет JS-памятью.
- **Node.js** - runtime, который добавляет к JavaScript системные API.
- **libuv** - библиотека event loop, async I/O и thread pool.
- **Event loop** - цикл обработки задач, событий и callbacks.
- **Event multiplexer** - системный механизм ожидания множества событий.
- **Callback** - функция, которую вызывают позже.
- **Promise** - объект будущего результата.
- **Microtask** - очередь задач Promise и похожих механизмов.
- **Thread pool** - набор worker-потоков для блокирующих операций.
- **Worker thread** - отдельный поток для JavaScript-кода.
- **Heap** - область памяти для объектов.
- **Garbage collector** - механизм автоматического освобождения недостижимой
  памяти.

## 14. Финальная сцена

Node.js кажется магией, пока смешаны все уровни. Но если разделить роли, картина
становится ясной:

- V8 исполняет JavaScript.
- Node.js дает API.
- libuv организует event loop и работу с ОС.
- event multiplexer помогает ждать много событий одним потоком.
- thread pool и worker threads нужны там, где ожидание или вычисления могут
  блокировать основной поток.
- garbage collector освобождает недостижимую память, но не спасает от
  сохраненных лишних ссылок.
- безопасность строится не на одном механизме, а на дисциплине границ,
  валидации, минимальных прав и внимательном отношении к зависимостям.

Это и есть крепкий фундамент: понимать, какой слой за что отвечает, и не
ожидать от одного слоя работы другого.
